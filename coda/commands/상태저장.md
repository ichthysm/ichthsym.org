# 프로젝트 상태 저장

## 핵심 원칙
1. **일관성**: 기존 PROJECT_STATUS.md 구조/스타일 100% 준수
2. **연속성**: 다음 세션에서 바로 이어서 작업 가능하도록 인계 정보 필수
3. **간결성**: 최근 10개 버전만 유지, 이전 버전은 CHANGELOG.md로 이동

## 사전 확인 (필수)
1. 기존 `PROJECT_STATUS.md` 읽고 구조/스타일 파악
2. `CLAUDE.md` 참조하여 프로젝트 컨텍스트 확인
3. `VERSION` 파일로 현재 버전 확인

## 참조 문서 섹션 (헤더 아래 유지)
```markdown
## 참조 문서

| 문서 | 설명 |
|------|------|
| [CLAUDE.md](./CLAUDE.md) | 개발 가이드, 아키텍처, 코드 컨벤션 |
| [CHANGELOG.md](./CHANGELOG.md) | 전체 변경 이력 (아카이브) |
| [VERSION](./VERSION) | 현재 버전 (단일 소스) |
```

## 업데이트 내용

### 1. 헤더 업데이트
- 마지막 업데이트 날짜 (오늘)
- 현재 버전 (VERSION 파일 참조)
- 개발 브랜치

### 2. 최근 변경사항 추가
- 이번 세션 작업 내용을 **기존 스타일 그대로** 추가
- 버전 태그: `### v{VERSION} - 제목 완료 (날짜)`
- Backend/Frontend/기타 변경사항 분류
- 담당자 페르소나 표기 (박안도, 유아이 등)
- 구체적 파일명, 기능명 포함

### 3. 세션 인계 섹션 (필수)
기존에 없으면 문서 마지막에 추가:

```markdown
---

## 현재 진행 상황 (세션 인계용)

### 마지막 작업
- 수행한 작업: [구체적 내용]
- 수정한 파일: [파일 목록]
- 커밋 여부: 완료/미완료

### 진행 중 작업 (미완료)
- [ ] 작업 내용
- 막힌 부분: [있으면 기술]
- 다음 단계: [구체적 액션]

### 다음 세션 TODO
1. [우선순위 1]
2. [우선순위 2]
```

## 파일 크기 관리 (중요)
PROJECT_STATUS.md가 10개 버전 초과 시:
1. 오래된 버전을 CHANGELOG.md로 이동
2. 상세 내용은 CHANGELOG.md 참조하도록 링크 유지

## 금지 사항
- 기존 구조 임의 변경 금지
- 섹션 순서 변경 금지
- 이전 버전 변경사항 수정 금지
- PROJECT_STATUS.md가 2,000줄 초과하지 않도록 관리

## 마무리 (필수)
상태 저장 완료 후 반드시 커밋 및 푸시:

```bash
git add PROJECT_STATUS.md
git commit -m "docs: 프로젝트 상태 업데이트 (v{VERSION})"
git push
```
