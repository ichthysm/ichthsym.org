# 병렬 작업 실행 프로토콜

병렬 서브에이전트 실행 시 중간보고 및 완료 수집 규칙입니다.

## 실행 방식

- 모든 독립 작업은 `run_in_background: true`로 실행
- 각 에이전트 프롬프트 끝에 [진행 보고 규칙] 블록 포함

## 서브에이전트 컨텍스트 프리픽스 (필수)

Task() 호출 시, 프롬프트 **상단에** 해당 역할의 memory/ 파일 핵심 내용을 포함:

| 에이전트 역할 | 참조 파일 |
|---------------|-----------|
| Backend (박안도) | `memory/backend-patterns.md` + `memory/common-pitfalls.md` |
| Frontend (유아이) | `memory/frontend-patterns.md` + `memory/common-pitfalls.md` |
| Security (Chloe) | `memory/common-pitfalls.md` |
| 기타 | `memory/common-pitfalls.md` |

프롬프트 구조:
```
[프로젝트 컨텍스트]
- {프로젝트명} v{VERSION}, {기술 스택}
- {memory/ 파일에서 해당 작업과 관련된 주의사항 발췌}

[작업 지시]
{실제 작업 내용}

[진행 보고 규칙]
...
```

## 서브에이전트 진행 보고 템플릿

작업 프롬프트 끝에 아래 블록을 항상 포함:

```
[진행 보고 규칙]
작업 시작/중간/완료 시 아래 형식으로 출력하라:

- 시작:
[START] {담당자명} - {작업 제목} ({대상 파일 수}건)

- 단계별 진행:
[PROGRESS] {N}/{전체} - {현재 파일명 또는 작업} - {상태: 분석중|수정중|완료}

- 완료:
[DONE] {담당자명} - 수정 {N}건, 신규 {N}건, 삭제 {N}건
[FILES] {변경된 파일 목록, 쉼표 구분}
[SUMMARY] {1줄 요약}

- 오류/차단 시:
[ERROR] {파일명}:{라인} - {오류 내용}
[BLOCKED] {차단 사유} - {필요한 의사결정}
```

## PM 중간 확인 패턴

캡틴이 "상황 보고", "진행 상황", "현황" 등을 요청하면:

1. 각 백그라운드 에이전트의 `TaskOutput`을 `block: false`로 조회
2. `[PROGRESS]` 태그를 파싱하여 종합 보고:

```
| 팀 | 담당 | 진행률 | 현재 작업 | 상태 |
|----|------|--------|----------|------|
| Frontend | 유아이 | 3/5 | Dashboard.tsx 수정중 | 진행중 |
| Backend | 박안도 | 1/3 | service.ts 분석중 | 진행중 |
| Security | Chloe | 완료 | JWT 모듈 감사 | 완료 |
```

3. `[BLOCKED]` 태그 발견 시 캡틴에게 의사결정 요청
4. `[ERROR]` 태그 발견 시 즉시 보고

## 완료 수집 패턴

모든 에이전트 완료 후:
1. 각 `[FILES]` 태그에서 변경 파일 목록 수집
2. 파일 충돌 확인 (동일 파일 수정 여부)
3. 충돌 없으면 → 종합 보고 + 빌드 확인
4. 충돌 있으면 → 캡틴에게 머지 방향 확인

## 에이전트별 max_turns 가이드

| 작업 규모 | max_turns | 예시 |
|----------|-----------|------|
| 분석만 (읽기) | 15 | 코드 리뷰, 보안 감사 |
| 소규모 수정 (1~3파일) | 20 | 버그 수정, 설정 변경 |
| 중규모 구현 (4~8파일) | 30 | 모듈 추가, 리팩토링 |
| 대규모 구현 (9파일+) | 50 | 새 기능 풀스택 |
